# Quy chuẩn Thông điệp Commit (Sentry Style)

Tuân thủ các quy chuẩn này khi tạo commit cho các dự án. Toàn bộ thông điệp commit (subject và body) phải được viết bằng **Tiếng Việt**.

## Khi nào sử dụng
- Người dùng yêu cầu commit code, chuẩn bị thông điệp commit, hoặc lưu thay đổi trong git.
- Tác vụ yêu cầu định dạng commit kiểu Sentry với cấu trúc conventional commit và tham chiếu issue.
- Tác vụ yêu cầu kiểm tra an toàn nhánh trước khi commit, đặc biệt là tránh commit trực tiếp vào `main` hoặc `master`.

## Điều kiện tiên quyết

Trước khi commit, luôn luôn kiểm tra nhánh hiện tại:

```bash
git branch --show-current
```

**Nếu bạn đang ở nhánh `main` hoặc `master`, bạn BẮT BUỘC phải tạo một nhánh feature mới** — trừ khi người dùng yêu cầu rõ ràng việc commit vào main. Không hỏi người dùng có tạo nhánh hay không; chỉ cần tiến hành tạo nhánh. Skill `create-branch` vẫn sẽ đề xuất tên nhánh để người dùng xác nhận.

Sử dụng skill `create-branch` để tạo nhánh. Sau khi `create-branch` hoàn tất, hãy xác nhận nhánh hiện tại đã thay đổi trước khi tiếp tục:

```bash
git branch --show-current
```

Nếu vẫn ở `main` hoặc `master` (ví dụ: người dùng hủy tạo nhánh), hãy dừng lại — không commit.

## Định dạng

```
<type>(<scope>): <subject>

<body>

<footer>
```

Tiêu đề (header) là bắt buộc. Scope (phạm vi) là tùy chọn. Tất cả các dòng phải dưới 100 ký tự. Toàn bộ phần `<subject>` và `<body>` phải viết bằng **Tiếng Việt**.

## Các loại Commit (Types)

| Loại | Mục đích |
|------|---------|
| `feat` | Tính năng mới |
| `fix` | Sửa lỗi |
| `ref` | Tái cấu trúc (không thay đổi hành vi) |
| `perf` | Cải thiện hiệu năng |
| `docs` | Chỉ thay đổi tài liệu |
| `test` | Thêm hoặc sửa lỗi logic test |
| `build` | Hệ thống build hoặc dependencies |
| `ci` | Cấu hình CI |
| `chore` | Các tác vụ bảo trì định kỳ |
| `style` | Định dạng code (không thay đổi logic) |
| `meta` | Metadata của repository |
| `license` | Thay đổi giấy phép |

## Quy tắc dòng Tiêu đề (Subject)

- Sử dụng động từ ở thì hiện tại/mệnh lệnh: "Thêm tính năng" thay vì "Đã thêm tính năng"
- Viết hoa chữ cái đầu tiên
- Không để dấu chấm ở cuối
- Tối đa 70 ký tự
- **Bắt buộc viết bằng Tiếng Việt**

## Hướng dẫn phần Thân bài (Body)

- Giải thích **cái gì** và **tại sao**, đừng giải thích như thế nào
- Sử dụng tiếng Việt rõ ràng, ngắn gọn
- Nêu rõ động lực cho sự thay đổi
- So sánh với hành vi cũ nếu cần thiết

## Footer: Tham chiếu Issue

Tham chiếu các issue ở phần footer sử dụng các mẫu sau:

```
Fixes GH-1234
Fixes #1234
Fixes SENTRY-1234
Refs LINEAR-ABC-123
```

- `Fixes` sẽ đóng issue khi được merge
- `Refs` chỉ liên kết mà không đóng issue

## Thay đổi do AI tạo ra (AI-Generated Changes)

Khi các thay đổi chủ yếu được tạo ra bởi một coding agent (như Antigravity), hãy bao gồm dòng Co-Authored-By ở footer của commit:

```
Co-Authored-By: Antigravity <noreply@google.com>
```

Đây là chỉ báo duy nhất về sự tham gia của AI xuất hiện trong commit. Không thêm các cụm từ như "Generated by AI", "Được viết bởi AI", hoặc các dấu hiệu tương tự trong subject, body, hoặc bất kỳ nơi nào khác.

## Ví dụ

### Sửa lỗi đơn giản

```
fix(api): Xử lý phản hồi null trong endpoint người dùng

API người dùng có thể trả về null đối với các tài khoản đã xóa, gây ra lỗi
crash trên bảng điều khiển. Thêm kiểm tra null trước khi truy cập thuộc tính.

Fixes SENTRY-5678
Co-Authored-By: Antigravity <noreply@google.com>
```

### Tính năng có scope

```
feat(alerts): Thêm phản hồi theo luồng Slack cho các cập nhật cảnh báo

Khi một cảnh báo được cập nhật hoặc giải quyết, hãy gửi phản hồi vào luồng
Slack gốc thay vì tạo tin nhắn mới. Điều này giúp các thông báo liên quan
được nhóm lại với nhau.

Refs GH-1234
```

### Tái cấu trúc (Refactor)

```
ref: Tách logic xác thực chung vào module dùng chung

Di chuyển mã xác thực trùng lặp từ ba endpoint vào một lớp validator
dùng chung. Không thay đổi hành vi của hệ thống.
```

### Thay đổi gây phá vỡ (Breaking change)

```
feat(api)!: Loại bỏ các endpoint v1 đã lỗi thời

Xóa tất cả các endpoint API v1 đã bị đánh dấu lỗi thời từ phiên bản 23.1.
Các client nên chuyển sang sử dụng các endpoint v2.

BREAKING CHANGE: Các endpoint v1 không còn khả dụng
Fixes SENTRY-9999
```

## Định dạng Revert

```
revert: feat(api): Thêm endpoint mới

Hoàn tác commit abc123def456.

Lý do: Gây ra suy giảm hiệu năng trên môi trường production.
```

## Nguyên tắc

- Mỗi commit nên là một thay đổi đơn lẻ, ổn định
- Các commit nên có khả năng review độc lập
- Repository phải luôn ở trạng thái hoạt động được sau mỗi commit
